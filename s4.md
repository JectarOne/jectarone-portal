# Sprint 5 — Vulnerability Management Workflow

## Goal

Transform JectarOne from a reporting platform into a complete Vulnerability Management platform.

The objective of this sprint is to allow organizations to track, assign, validate, and close vulnerabilities throughout their entire lifecycle.

---

# Requirements

## 1. Vulnerability Status Workflow

Replace the current simple status with a complete workflow.

Supported statuses:

* Open
* In Progress
* Ready for Validation
* Resolved
* Accepted Risk
* False Positive

Requirements:

* Status history must be preserved.
* Every status change must record:

  * Previous status
  * New status
  * User
  * Timestamp

---

## 2. Finding Assignment

Each finding can be assigned to a user within the organization.

Add:

* Assignee
* Assigned date
* Assigned by

Requirements:

* Only users belonging to the same organization can be assigned.
* Reassignment must be tracked in the activity history.

---

## 3. SLA Management

Each severity automatically receives a remediation deadline.

Default SLA:

| Severity      | Due     |
| ------------- | ------- |
| Critical      | 7 days  |
| High          | 30 days |
| Medium        | 60 days |
| Low           | 90 days |
| Informational | No SLA  |

Requirements:

* Automatically calculate due date.
* Allow manual override.
* Display overdue findings.
* Dashboard counters for SLA breaches.

---

## 4. Activity Timeline

Every important action creates an immutable timeline event.

Examples:

* Finding created
* Severity changed
* Status changed
* Assigned
* Comment added
* Evidence uploaded
* Risk accepted
* Validation completed

Each event contains:

* User
* Timestamp
* Action
* Metadata

Display the timeline in chronological order.

---

## 5. Comments

Allow collaboration.

Each finding supports:

* Comments
* Edit own comment
* Delete own comment (soft delete)
* Mention users
* Markdown formatting

---

## 6. Evidence Management

Support evidence attachments.

Accepted formats:

* PNG
* JPG
* PDF
* TXT
* ZIP

Store:

* File name
* Size
* Uploaded by
* Uploaded at

Requirements:

* Organization isolation
* Soft delete
* Download
* Preview images

---

## 7. Risk Acceptance

Organizations may decide not to remediate a finding.

Add:

* Accepted by
* Business justification
* Expiration date
* Review reminder

Workflow:

Open

↓

Accepted Risk

↓

Reopen (optional)

Accepted risks should appear separately on dashboards.

---

## 8. Finding Filters

Support filtering by:

* Asset
* Severity
* Status
* Assignee
* CVSS
* CWE
* Created date
* Due date
* Overdue
* Accepted Risk

Filters should be combinable.

---

## 9. Dashboard Improvements

Create executive metrics.

Include:

* Open Findings
* Closed Findings
* Overdue Findings
* Accepted Risks
* Critical Findings
* High Findings
* Average CVSS
* MTTR (Mean Time To Resolve)
* Findings created this month
* Findings resolved this month

Charts:

* Severity distribution
* Status distribution
* Findings over time
* Resolution trend

---

## 10. Search

Global search across:

* Finding title
* Description
* Asset
* CVE
* CWE
* CVSS
* Comments

Must be organization scoped.
